## 6. menu_cache_offline_fallback | medium | Cashier sees cached menu when offline; updates when online

### 1. Identification
- **Name**: `menu_cache_offline_fallback`
- **Complexity**: `medium`
- **Source**: `src/hooks/useMenu.js`. Template sections ‚ÄúCASHIER SCREEN‚Äù and ‚ÄúDATA STORAGE ARCHITECTURE ‚Äì Menu cache in local IndexedDB‚Äù.

### 2. Condition fulfilled
- **User need**: The cashier can see the full menu even when the device is offline, ensuring that order entry is never blocked by network hiccups. When online, menu changes from the owner appear automatically.
- **Exact technical condition**: On cashier login, the app first reads the menu from the local `menuCache` IndexedDB store and displays it immediately. Simultaneously, it sets up a Firestore `onSnapshot` listener on the `menu` collection. When the snapshot receives data (online), it updates the React state and also writes the fresh menu to `menuCache`, overwriting the cache. When offline, the listener fails silently (or reconnects later) and the cached menu remains visible.

### 3. Implementation pattern
- **Key technologies / APIs**: IndexedDB (store `menuCache`), Firestore `onSnapshot`, React `useEffect` with cleanup.
- **Core code sketch** (minimal):
  ```js
  useEffect(() => {
    let unsubscribe = null;
    async function loadFromCache() {
      const db = await openDB();
      const cached = await db.transaction("menuCache").objectStore("menuCache").getAll();
      if (cached.length) setMenuItems(cached);
    }
    loadFromCache();
    const q = query(collection(db, "menu"), where("shopId", "==", SHOP_ID));
    unsubscribe = onSnapshot(q, async (snapshot) => {
      const items = snapshot.docs.map(doc => ({ itemId: doc.id, ...doc.data() }));
      setMenuItems(items);
      // update cache
      const cacheDb = await openDB();
      const tx = cacheDb.transaction("menuCache", "readwrite");
      await tx.objectStore("menuCache").clear();
      for (const item of items) await tx.objectStore("menuCache").add(item);
      await tx.done;
    });
    return () => unsubscribe && unsubscribe();
  }, []);
  ```
- **File structure / hooks**:
  - `hooks/useMenu.js` ‚Äì encapsulates the cache‚Äëthen‚Äënetwork logic.
  - `db/indexedDB.js` ‚Äì defines `menuCache` store with indexes on `category` and `outOfStock`.
  - `cashier/CashierScreen.jsx` ‚Äì consumes `useMenu`.

### 5. Known limits and edge cases
- **What it does NOT handle**:  
  - If the cache is empty and the device is offline on first launch, the cashier screen will show ‚ÄúNo menu items‚Äù (because there‚Äôs no cache yet). The template expects the owner to have added items before cashier use.  
  - Menu changes made by the owner while the cashier is offline will not be visible until connectivity is restored and the `onSnapshot` updates the cache.  
  - The cache is overwritten entirely on each snapshot; this is fine for small menus.
- **Fails gracefully**: If the Firestore listener fails (e.g., permission denied), it falls back to the cache and shows a stale menu; no crash. If IndexedDB read fails, the component shows loading state and then attempts network fetch.
- **Scalability ceilings**: The menu is expected to have <100 items, so storing/retrieving all items at once is fine.

  - The cache may be empty on first run ‚Äì not a bug, by design.
