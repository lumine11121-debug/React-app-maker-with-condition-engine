# FULFILLER TEMPLATE

### 1. Name, version, complexity & purpose
- **Name**: sequential_order_number
- **Version**: v1
- **Complexity**: medium
- **Purpose**: Generate sequential order numbers that persist across sessions and incorporate pending offline orders.

### 2. Trigger conditions (ALL must be satisfied; use OR for alternatives)
- Orders require human‑readable sequential numbering: Yes
- Single cashier assumption is acceptable: Yes

### 3. When NOT to choose (disqualifiers)
- Multiple cashiers concurrently (collisions possible) → disqualify.
- Requires global atomic counter (e.g., distributed system) → disqualify (v1 not designed for it).

### 4. Dependencies (all required; OR allowed)
- firebase_initialisation (to query max orderNumber)
- indexed_db_setup (writeQueue and orders stores)

### 5. Edge cases / known limits (including graceful failure)
- If remote query fails, falls back to `max = 0` and returns 1, logging error.
- If write queue has very high number from another device, may skip numbers but never reuse.
- Graceful failure: numbers may reset to 1, but ordering continues.

### 6. Verification steps (user test)
1. Create orders online → numbers increment (1,2,3…).
2. Go offline, create orders → numbers continue (4,5…).
3. Go online, flush → same numbers appear in remote DB.
4. Close and reopen cashier → next number continues from highest seen.

### 7. Implementation sketch
Optional (medium). Pattern:
```js
const maxRemote = await getMaxOrderNumberFromFirebase();
const maxQueue = await getMaxFromWriteQueue();
const maxLocal = await getMaxFromLocalOrders();
const next = Math.max(maxRemote, maxQueue, maxLocal) + 1;
