## 2. realtime_firestore_sync | medium | Live updates for today‚Äôs data (owner dashboard, day period)

### 1. Identification
- **Name**: `realtime_firestore_sync`
- **Complexity**: `medium`
- **Source**: `src/hooks/useOrders.js` (when `period === "day"` and `isLive === true`), `src/owner/MiddleTab.jsx` (passes `isLive=true` for day period). Template section ‚ÄúREAL-TIME DATA‚Äù and ‚ÄúMiddle Tab (Daily Essentials)‚Äù.

### 2. Condition fulfilled
- **User need**: Owner sees current day‚Äôs revenue, orders, and splits update automatically as cashiers confirm orders, without refreshing the page.
- **Exact technical condition**: When the owner selects ‚ÄúDay‚Äù period in the dashboard, a Firestore `onSnapshot` listener is attached to the `orders` collection with a filter `timestamp >= todayStart` and `shopId == SHOP_ID`. The listener sends real‚Äëtime updates to the UI. All five dashboard sections are derived from the same snapshot.

### 3. Implementation pattern
- **Key technologies / APIs**: Firestore `onSnapshot`, `query`, `where`, `orderBy`, `Timestamp`. React `useEffect` for setup and cleanup.
- **Core code sketch** (minimal):
  ```js
  const todayStart = new Date(); todayStart.setHours(0,0,0,0);
  const startTimestamp = Timestamp.fromDate(todayStart);
  const q = query(
    collection(db, "orders"),
    where("shopId", "==", SHOP_ID),
    where("timestamp", ">=", startTimestamp),
    orderBy("timestamp", "asc")
  );
  const unsubscribe = onSnapshot(q, (snapshot) => {
    const orders = snapshot.docs.map(doc => ({ orderId: doc.id, ...doc.data() }));
    const derived = deriveAllSections(orders);
    setData(derived);
  });
  return () => unsubscribe();
  ```
- **File structure / hooks**:
  - `hooks/useOrders.js` ‚Äì contains the `onSnapshot` logic when `period === "day" && isLive === true`.
  - `owner/MiddleTab.jsx` ‚Äì calls `useOrders(globalPeriod, globalPeriod === "day")`.
  - `utils/chartHelpers.js` ‚Äì `deriveAllSections(orders)` computes all five dashboard sections from a single order array.

### 4. Setup commands (if any)
- **Install**: Already part of Firebase SDK (`npm install firebase`).
- **Initialisation**: Firestore must be enabled and security rules allow reads. Composite index on `shopId` (asc) + `timestamp` (asc) is required (see template prerequisites).
- **Browser/API permissions**: No special permissions; relies on Firestore SDK.

### 5. Known limits and edge cases
- **What it does NOT handle**:  
  - Does not work offline (relies on Firestore connection). Offline periods show stale data; the app shows offline banner instead.  
  - Does not paginate large day volumes (assumes <1000 orders per day).  
  - Listener remains active while day tab is open; switching to Week/Month stops the listener.
- **Fails gracefully**: If network drops, `onSnapshot` will fire an error and the hook will set `error` state; the UI shows an error message but does not crash.
- **Scalability ceilings**: Firestore free tier read limit (50k/day). Day queries typically cost 1 read per order (plus initial snapshot). Fine for normal daily volume.

### 6. Dependencies on other fulfillers
- **Depends on**:  
  - `firebase_initialisation` ‚Äì Firestore must be configured.  
  - `firebase_server_timestamp` ‚Äì orders must use Server Timestamp for correct date filtering.  
  - `deriveAllSections` helper (from `chartHelpers.js`).
- **Must be present before use**:  
  - Firestore database created and security rules allow reads.  
  - Composite index on `shopId` + `timestamp` created (otherwise `onSnapshot` throws an error).

### 7. Verification / test
- **Human verification**:  
  1. Open owner dashboard, select ‚ÄúDay‚Äù period.  
  2. On cashier device, create a new order and confirm.  
  3. Within a few seconds, the owner dashboard should update automatically (total revenue, order count, etc.).  
  4. Open another browser tab with the same owner dashboard; it should also update live.  
  5. Switch to ‚ÄúWeek‚Äù period ‚Äì live updates stop (no new orders appear until you refresh or switch back to Day).  
  6. Disable network ‚Äì dashboard will show an error banner; re‚Äëenable network ‚Äì listener reconnects and updates again.
