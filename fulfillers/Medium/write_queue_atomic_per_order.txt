## 11. write_queue_atomic_per_order | medium | Flush queue: each order written to Firebase, then local orders, then delete from queue atomically

### 1. Identification
- **Name**: `write_queue_atomic_per_order`
- **Complexity**: `medium`
- **Source**: `src/db/queueService.js` (flushQueue function). Template section “WRITE QUEUE FLUSH” and “ATOMICITY IS NON-NEGOTIABLE”.

### 2. Condition fulfilled
- **User need**: When the device reconnects, queued offline orders are reliably saved to Firebase and to the local cache, without duplicates or partial saves. If any step fails for one order, that order stays in the queue for later retry, and other orders continue.
- **Exact technical condition**: For each order in the write queue, the `flushQueue` function first writes the order to Firestore using `saveOrder`. On success, it opens an IndexedDB transaction that writes the order to the local `orders` store and deletes the same order from the `writeQueue` store atomically. The two operations (local `orders` put + `writeQueue` delete) are wrapped in a single transaction – either both commit or both abort. If the Firebase write fails, the order remains in the queue and no local changes are made.

### 3. Implementation pattern
- **Key technologies / APIs**: Firestore `addDoc`, IndexedDB transaction covering two object stores (`orders` and `writeQueue`).
- **Core code sketch** (minimal):
  ```js
  async function flushQueue(onProgress) {
    const queue = await getQueue();
    let success = 0, failed = 0;
    for (const item of queue) {
      try {
        const saved = await saveOrder(item.orderData);
        const db = await openDB();
        const tx = db.transaction(["orders", "writeQueue"], "readwrite");
        const ordersStore = tx.objectStore("orders");
        await ordersStore.put(saved);
        const queueStore = tx.objectStore("writeQueue");
        await queueStore.delete(item.orderId);
        await new Promise((resolve, reject) => {
          tx.oncomplete = resolve;
          tx.onerror = () => reject(tx.error);
        });
        db.close();
        success++;
      } catch (err) {
        console.error("Failed to flush order", item.orderId, err);
        failed++;
      }
    }
    return { success, failed };
  }
  ```
- **File structure / hooks**:
  - `db/queueService.js` – `flushQueue` and helper functions.
  - `hooks/useWriteQueue.js` – calls `flushQueue` on online event and periodically.

- **Install**: None; relies on existing IndexedDB and Firebase.

- **Browser/API permissions**: None.

### 5. Known limits and edge cases
- **What it does NOT handle**:  
  - If `saveOrder` partially writes (e.g., network timeout but document actually created), the order may be duplicated on subsequent flush. Firestore document IDs are generated client‑side, so the same order could be written twice. In practice, this is rare and acceptable for v1. A future improvement would check for existing `orderNumber` before writing.  
  - Does not retry a failed Firebase write with exponential backoff (only retries on next flush attempt).  
  - If the IndexedDB transaction fails (e.g., quota exceeded), the order remains in the queue, but the user may have already received a success toast – this is a known but unlikely edge case.
- **Fails gracefully**: For any failure, the order stays in the queue, and the rest of the queue continues. No data loss. The offline banner persists.
- **Scalability ceilings**: Loops through all queued orders sequentially; for thousands of orders, flush may be slow but still eventually completes.

### 7. Verification / test
- **Human verification**:  
  1. Go offline, create 2‑3 orders.  
  2. Open browser DevTools → Application → IndexedDB → `writeQueue` – verify orders are present.  
  3. Go online. Observe the flush progress (badge decreases, orders appear in Firebase).  
  4. Immediately after flush, check `writeQueue` – should be empty.  
  5. Check `orders` store in IndexedDB – the same orders should be present (cached).  
  6. Simulate a Firebase write error (e.g., temporarily block network after first order). The remaining orders stay in queue.  
  7. Restore network – flush continues.
