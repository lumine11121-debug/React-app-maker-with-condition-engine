## 13. cache_then_network_menu | medium | Menu fetch: show cached immediately, then update from network when available

### 1. Identification
- **Name**: `cache_then_network_menu`
- **Complexity**: `medium`
- **Source**: `src/hooks/useMenu.js`. Template sections “MENU CACHE – Cashier side” and “FIXED IDEAS – Cashier screen”.

### 2. Condition fulfilled
- **User need**: The cashier sees the menu instantly on app open, even on slow or flaky networks, and the menu updates automatically when changes are made by the owner (e.g., out of stock toggle, price edit). This ensures no blank screen waiting for network and no stale data when online.
- **Exact technical condition**: On cashier login, the hook first reads the menu from the local `menuCache` IndexedDB store and renders it immediately. Simultaneously, it sets up a Firestore `onSnapshot` listener on the `menu` collection. When the snapshot data arrives (online), it updates the React state and also writes the fresh menu to `menuCache`, overwriting the cache. If offline, the listener may fail or stay pending, but the cached menu remains visible. The listener is cancelled on unmount.

### 3. Implementation pattern
- **Key technologies / APIs**: IndexedDB (`menuCache` store), Firestore `onSnapshot`, React `useEffect` with cleanup.
- **Core code sketch** (minimal):
  ```js
  useEffect(() => {
    let unsubscribe = null;
    async function loadCache() {
      const db = await openDB();
      const cached = await db.transaction("menuCache").objectStore("menuCache").getAll();
      if (cached.length) setMenuItems(cached);
      setLoading(false);
    }
    loadCache();
    const q = query(collection(db, "menu"), where("shopId", "==", SHOP_ID));
    unsubscribe = onSnapshot(q, async (snapshot) => {
      const items = snapshot.docs.map(doc => ({ itemId: doc.id, ...doc.data() }));
      setMenuItems(items);
      // update cache
      const cacheDb = await openDB();
      const tx = cacheDb.transaction("menuCache", "readwrite");
      const store = tx.objectStore("menuCache");
      await store.clear();
      for (const item of items) await store.add(item);
      await tx.done;
    });
    return () => unsubscribe && unsubscribe();
  }, []);
  ```
- **File structure / hooks**:
  - `hooks/useMenu.js` – encapsulates the logic.
  - `db/indexedDB.js` – defines `menuCache` store with indexes.
  - `cashier/CashierScreen.jsx` – uses `useMenu`.

### 4. Setup commands (if any)
- **Install**: None; uses existing IndexedDB and Firebase.
- **Initialisation**: IndexedDB database must include `menuCache` store (created in `onupgradeneeded`).
- **Browser/API permissions**: None.

### 5. Known limits and edge cases
- **What it does NOT handle**:  
  - If the cache is empty on first launch and the device is offline, the cashier screen will show “No menu items” (because no network fetch yet). The template expects the owner to have added items before cashier use, or the first fetch will populate the cache.  
  - Menu updates made by the owner while the cashier is offline will not be visible until connectivity restores and the `onSnapshot` updates the cache. This is documented as acceptable.  
  - The cache is completely overwritten on each snapshot; for large menus this is fine.
- **Fails gracefully**: If the Firestore listener fails (e.g., permission denied), the cache remains and the menu stays usable (though stale). If IndexedDB read fails, the hook falls back to loading state and then attempts network fetch; if both fail, the menu is empty.
- **Scalability ceilings**: The menu is expected to have <100 items; storing/retrieving all items at once is efficient.

### 6. Dependencies on other fulfillers
- **Depends on**:  
  - `indexed_db_setup` – must have `menuCache` store.  
  - `firebase_initialisation` – Firestore must be configured with `menu` collection.
- **Must be present before use**:  
  - The menu collection in Firestore must have at least some items (owner must add them).  
  - On first run, the cache may be empty – not a bug, by design.

### 7. Verification / test
- **Human verification**:  
  1. Open cashier screen while online – menu appears (populates cache).  
  2. Go offline, refresh cashier screen – menu appears from cache (no network requests).  
  3. While offline, order items – menu works.  
  4. While cashier is offline, as owner, toggle an item’s out of stock status.  
  5. Go online on cashier – the menu should update to reflect the change (listener delivers new data, cache updated).  
  6. Clear IndexedDB, then open cashier offline – “No menu items” appears (expected).  
  7. Go online – menu loads and cache is repopulated.
