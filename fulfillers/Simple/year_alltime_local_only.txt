## 5. year_alltime_local_only | simple | Year/All Time summaries read from local IndexedDB only, zero Firebase reads

### 1. Identification
- **Name**: `year_alltime_local_only`
- **Complexity**: `simple`
- **Source**: `src/owner/YearSummary.jsx`, `src/owner/AllTimeSummary.jsx`, `src/hooks/useOrders.js` (when `period === "year"` or `period === "alltime"` with `isLive=false`). Template section “Year and All Time as dedicated screens”.

### 2. Condition fulfilled
- **User need**: Owner can view aggregated sales data for the entire year or for all time without consuming Firebase read quota. The data is derived from the local IndexedDB cache that is already synced.
- **Exact technical condition**: When the owner navigates to the Year or All Time screen, the `useOrders` hook is called with `period="year"` (or `"alltime"`) and `isLive=false`. The hook reads all orders from the local `orders` IndexedDB store, then filters by the period (for Year) and derives all five dashboard sections (revenue, order count, payment split, items performance, sales by hour). No Firestore queries are made. The UI prominently displays a “Last synced” timestamp to indicate data currency.

### 3. Implementation pattern
- **Key technologies / APIs**: IndexedDB (via `openDB` and `getAll()` on `orders` store), JavaScript array filtering and aggregation.
- **Core code sketch** (minimal):
  ```js
  // In useOrders.js for period "year" and isLive=false
  const allOrders = await db.transaction("orders").objectStore("orders").getAll();
  const now = new Date();
  const startDate = new Date(now.getFullYear(), 0, 1);
  const filtered = allOrders.filter(order => order.timestamp?.toDate() >= startDate);
  const derived = deriveAllSections(filtered);
  return derived;
  ```
- **File structure / hooks**:
  - `owner/YearSummary.jsx` – uses `useOrders("year", false)` and renders the same five sections as `MiddleTab`.
  - `owner/AllTimeSummary.jsx` – uses `useOrders("alltime", false)`.
  - `hooks/useOrders.js` – contains the IndexedDB read logic for historical periods.
  - `utils/chartHelpers.js` – `deriveAllSections(orders)`.

### 4. Setup commands (if any)
- **Install**: None beyond existing IndexedDB and React.
- **Initialisation**: IndexedDB `orders` store must be populated by first‑install sync and incremental sync.
- **Browser/API permissions**: None.

### 5. Known limits and edge cases
- **What it does NOT handle**:  
  - Does not update in real time when new orders are added (owner must re‑enter the screen or refresh to see updated data).  
  - No pagination; loading all orders into memory may be slow for very large datasets (>10k orders).  
  - If first‑install sync hasn’t completed, the screen may show empty or incomplete data.
- **Fails gracefully**: If IndexedDB `orders` store is empty, the screen shows zero values and a “No items sold” message. If the store is missing, an error is caught and displayed.
- **Scalability ceilings**: Performance degrades with number of orders; for >10k orders, aggregation may cause UI lag. Future version could add pagination or background worker.

### 6. Dependencies on other fulfillers
- **Depends on**:  
  - `indexed_db_setup` – must have `orders` store with `timestamp` index.  
  - `first_install_bulk_sync` and `incremental_sync_indexeddb` – the local `orders` store must be populated.
- **Must be present before use**:  
  - `firstSyncComplete` should be true (local cache contains at least some data).  
  - `orders` store must be readable.

### 7. Verification / test
- **Human verification**:  
  1. Ensure first‑install sync has completed (owner dashboard loaded at least once).  
  2. On owner dashboard, click the “YEAR” button.  
  3. The Year Summary screen should appear, showing aggregated data for the current year.  
  4. Verify that no Firestore requests are made (open browser Network tab – no `firestore.googleapis.com` requests).  
  5. Click “ALL TIME” button – similar verification, no network requests.  
  6. Check “Last synced” timestamp – it should reflect the last time the owner opened the dashboard (or last incremental sync).  
  7. On cashier, add a new order. Then re‑enter Year/All Time screens – data should update (after the next sync, when owner reopens the dashboard or refreshes).
