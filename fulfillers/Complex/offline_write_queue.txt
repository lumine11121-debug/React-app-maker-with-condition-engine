
### 1. Identification
- **Name**: `offline_write_queue`
- **Complexity**: `complex`
- **Source**: `src/db/queueService.js`, `src/hooks/useWriteQueue.js`, `src/cashier/OrderSummary.jsx`, `src/cashier/CashierScreen.jsx`; template sections “3. ORDER SUMMARY SCREEN” and “5. DATA STORAGE ARCHITECTURE”.

### 2. Condition fulfilled
- **User need**: Cashiers can confirm orders even when the device is offline; orders are saved locally and automatically sync when the internet connection returns, without manual intervention.
- **Exact technical condition**: Orders created while `navigator.onLine === false` are stored in an IndexedDB `writeQueue` store. When the browser fires the `online` event (or a periodic check detects connectivity), the queue is flushed: each order is saved to Firestore, then added to the local `orders` cache, and removed from the queue. The operation is atomic per order.

### 3. Implementation pattern
- **Key technologies / APIs**: IndexedDB (via custom wrapper `openDB`), `window.online` / `window.offline` events, `navigator.onLine`, Firestore `addDoc`, `Promise` wrappers for IndexedDB async operations.
- **Core code sketch** (minimal):
  ```js
  // addToQueue
  const queueItem = { orderId: `local_${Date.now()}_${Math.random()}`, orderData, queuedAt: new Date() };
  await db.transaction("writeQueue", "readwrite").objectStore("writeQueue").add(queueItem);
  
  // flushQueue
  const queue = await getQueue(); // uses getAll() with Promise
  for (const item of queue) {
    const saved = await saveOrder(item.orderData);
    await db.transaction("orders", "readwrite").objectStore("orders").put(saved);
    await db.transaction("writeQueue", "readwrite").objectStore("writeQueue").delete(item.orderId);
  }
  ```
- **File structure / hooks**:
  - `db/queueService.js` – `addToQueue`, `getQueue`, `removeFromQueue`, `flushQueue`.
  - `hooks/useWriteQueue.js` – exposes `queueLength`, `isOffline`, `flush`, `refreshQueueLength`. Listens to `online`/`offline` events and periodically flushes.
  - `cashier/OrderSummary.jsx` – uses `isOffline` prop to decide between `saveOrder` (online) and `addToQueue` (offline).
  - `cashier/CashierScreen.jsx` – passes `isOffline` and `refreshQueueLength` to `OrderSummary`.

### 4. Setup commands (if any)
- **Install**: Already part of project – no extra package. `npm install firebase` was already done.
- **Initialisation**: IndexedDB database is opened and stores created by `db/indexedDB.js` (version upgrade handles `writeQueue`). No extra init.

### 5. Known limits and edge cases
- **What it does NOT handle**:  
  - Conflict resolution if the same order is queued from multiple devices (single cashier assumed).  
  - Automatic retry with exponential backoff (only simple retry on each flush attempt).  
  - Large queues (>10k items) may cause slow flush and UI blocking (no batching in flush).
- **Fails gracefully**: If an individual order fails to save to Firestore, it remains in the queue and is retried on the next flush. The rest of the queue continues. The offline banner remains.
- **Scalability ceilings**: Queue stored in IndexedDB – practical limit is hundreds of MB; performance degrades with thousands of items.

### 6. Dependencies on other fulfillers

  - Firebase config must be valid.

### 7. Verification / test
- **Human verification**:  
  1. Go offline (airplane mode or DevTools offline).  
  2. Create an order on cashier screen → confirm → observe “Order saved locally” toast and queue badge (e.g., “📦 1 order(s) waiting to sync”).  
  3. Go online → within 30 seconds (or immediately if `online` event fires), badge disappears and order appears in Firebase console (`orders` collection) and on owner dashboard.  
  4. Repeat with browser closed while offline → reopen (still offline) → order should reappear in queue. Then go online → flush.
