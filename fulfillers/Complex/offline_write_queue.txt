# FULFILLER TEMPLATE

### 1. Name, version, complexity & purpose
- **Name**: offline_write_queue
- **Version**: v1
- **Complexity**: complex
- **Purpose**: Save orders when offline and automatically sync them to Firebase when connection returns.

### 2. Trigger conditions (ALL must be satisfied; use OR for alternatives)
- Offline record creation required: Yes
- (Firebase as source of truth) OR (Supabase as source): Yes
- User expects automatic sync on reconnect: Yes

### 3. When NOT to choose (disqualifiers)
- Multi‑device conflict resolution required (e.g., two cashiers offline simultaneously) → disqualify. (v1 assumes single cashier.)
- User requires real‑time bidirectional sync (e.g., collaborative editing) → disqualify.

### 4. Dependencies (all required; OR allowed)
- indexed_db_setup (must have `writeQueue` store)
- (firebase_initialisation) OR (supabase_initialisation)
- saveOrder function (writes to remote DB)

### 5. Edge cases / known limits (including graceful failure)
- IndexedDB quota exceeded → queue stops, user sees error banner.
- Large queue (>10k items) → flush may block UI; no batching.
- If remote write fails (e.g., network but DB error), order stays in queue, retries on next flush.
- Graceful failure: User sees “Working offline” banner; orders never lost.

### 6. Verification steps (user test)
1. Go offline (airplane mode or DevTools offline).
2. Create an order → toast “Order saved locally” and queue badge appears.
3. Go online → badge disappears within 30 seconds, order appears in remote database and local cache.
4. Close browser while offline with queued orders → reopen, go online → queue flushes.

### 7. Implementation sketch
Required (complex). Key patterns:
- IndexedDB `writeQueue` store with `orderId` keyPath.
- `addToQueue`: store order with `queuedAt` timestamp.
- `flushQueue`: for each queued order, save to remote, then atomic transaction: put into local `orders` + delete from `writeQueue`.
- `useWriteQueue` hook listens to `online`/`offline` events and periodic fallback.
