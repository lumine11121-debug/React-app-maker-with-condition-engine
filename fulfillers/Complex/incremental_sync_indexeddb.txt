## 3. incremental_sync_indexeddb | complex | Owner app fetches only new orders since last sync, atomic transaction

### 1. Identification
- **Name**: `incremental_sync_indexeddb`
- **Complexity**: `complex`
- **Source**: `src/db/syncService.js` (`incrementalSync` function), `src/hooks/useSync.js` (calls it on owner app open). Template section ‚ÄúINCREMENTAL SYNC‚Äù and ‚ÄúsyncService.js‚Äù in file structure.

### 2. Condition fulfilled
- **User need**: Owner sees all historical orders without waiting for a full download each time; after first install, only new orders since last visit are fetched, keeping startup fast and minimizing Firebase reads.
- **Exact technical condition**: On each owner app open (after first sync), the app reads `lastSyncTimestamp` from IndexedDB `syncMeta` store. It queries Firestore for orders with `timestamp > lastSyncTimestamp` (ordered ascending). New orders are written to the local `orders` store inside an IndexedDB transaction, and `lastSyncTimestamp` is updated to the newest order‚Äôs timestamp. The operation is atomic: either all new orders and the timestamp are written, or none are.

### 3. Implementation pattern
- **Key technologies / APIs**: Firestore `getDocs` with `where("timestamp", ">", lastTimestamp)`, IndexedDB transactions, `Timestamp` comparison.
- **Core code sketch** (minimal):
  ```js
  async function incrementalSync() {
    const lastTimestamp = await getLastSyncTimestamp();
    if (!lastTimestamp) return { count: 0 };
    const newOrders = await getOrdersSince(lastTimestamp);
    if (newOrders.length === 0) return { count: 0 };
    const db = await openDB();
    const tx = db.transaction(["orders", "syncMeta"], "readwrite");
    const ordersStore = tx.objectStore("orders");
    for (const order of newOrders) {
      await ordersStore.put(order);
    }
    const newLastTimestamp = newOrders[newOrders.length - 1].timestamp;
    const syncStore = tx.objectStore("syncMeta");
    await syncStore.put({ id: "sync", lastSyncTimestamp: newLastTimestamp, ... });
    await new Promise(resolve => { tx.oncomplete = resolve; });
    return { count: newOrders.length };
  }
  ```
- **File structure / hooks**:
  - `db/syncService.js` ‚Äì `incrementalSync`, `getLastSyncTimestamp`, `updateSyncMeta`.
  - `hooks/useSync.js` ‚Äì calls `incrementalSync` on mount if `firstSyncComplete` is true.
  - `db/indexedDB.js` ‚Äì defines `orders` store with index on `timestamp`.

### 4. Setup commands (if any)
- **Install**: No extra packages; uses existing Firebase and IndexedDB.
- **Initialisation**: IndexedDB must be opened with `orders` store and `syncMeta` store. The first install sync sets `firstSyncComplete` and initial `lastSyncTimestamp`.
- **Browser/API permissions**: None; all client‚Äëside.

### 5. Known limits and edge cases
- **What it does NOT handle**:  
  - If the atomic transaction fails (e.g., disk full), the sync may be partially written; the code attempts to revert but does not have rollback. In practice, the transaction either commits or aborts, so it's safe.  
  - Does not handle very large batches (>500 orders) because `getOrdersSince` in `orderService.js` does not paginate. In v1, we assume small daily order volume.  
  - If the user has multiple devices, each device maintains its own `lastSyncTimestamp` and local orders store; they are independent and may have different sync points ‚Äì acceptable.
- **Fails gracefully**: If Firestore query fails, the function throws and sync is retried on next app open. No data is lost.
- **Scalability ceilings**: Each incremental sync reads exactly 1 document per new order. Over a year of daily use, total reads ~ 365 * dailyOrders.
