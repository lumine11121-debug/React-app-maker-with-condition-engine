# FULFILLER 

### 1. Name, version, complexity & purpose
- **Name**: incremental_sync_indexeddb
- **Version**: v1
- **Complexity**: complex
- **Purpose**: Fetch only new records since last sync, atomically store them locally, and update sync timestamp.

### 2. Trigger conditions (ALL must be satisfied; use OR for alternatives)
- Owner app needs historical data offline: Yes
- Remote database supports timestamp filters (Firestore, PostgreSQL, etc.): Yes
- Local IndexedDB cache exists: Yes

### 3. When NOT to choose (disqualifiers)
- User requires real‑time live updates only (no local cache) → disqualify.
- Cannot store sync timestamp (e.g., no persistent storage) → disqualify.

### 4. Dependencies (all required; OR allowed)
- indexed_db_setup (stores: `orders`, `syncMeta`)
- (firebase_initialisation) OR (generic timestamp‑queryable API)

### 5. Edge cases / known limits (including graceful failure)
- If transaction fails, no data is written; next open retries.
- No batching for >500 items (v1); may exceed Firestore query limits.
- Graceful failure: Sync error shown, but previous local data remains usable.

### 6. Verification steps (user test)
1. Perform first‑install sync (all orders).
2. Add new order remotely.
3. Reopen owner app → only the new order is fetched (network tab confirms).
4. Clear IndexedDB; reopen → first install runs again.

### 7. Implementation sketch
Required. Core logic:
```js
const lastTimestamp = await getLastSyncTimestamp();
const newOrders = await fetchOrdersSince(lastTimestamp);
const tx = db.transaction(["orders", "syncMeta"], "readwrite");
for (const order of newOrders) await ordersStore.put(order);
await syncStore.put({ id: "sync", lastSyncTimestamp: newOrders.at(-1).timestamp });
await tx.complete;
