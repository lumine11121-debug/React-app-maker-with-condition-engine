## 4. first_install_bulk_sync | complex | Full historical sync with pagination, resumable

### 1. Identification
- **Name**: `first_install_bulk_sync`
- **Complexity**: `complex`
- **Source**: `src/db/syncService.js` (`firstInstallSync` function), `src/hooks/useSync.js` (calls it when `firstSyncComplete` is false). Template section “FIRST INSTALL SYNC”.

### 2. Condition fulfilled
- **User need**: When the owner installs the app for the first time (or clears local data), all existing orders from Firebase are downloaded to the local IndexedDB cache, enabling fast historical queries and offline access to past data. The process shows progress and can resume if interrupted.
- **Exact technical condition**: On first app open (owner side), the app checks `firstSyncComplete` in `syncMeta`. If false, it fetches all orders from Firestore in batches (default 500 per batch), writes each batch atomically to the local `orders` store, updates `lastSyncTimestamp` after each batch, and finally sets `firstSyncComplete` to true. A loading screen blocks dashboard access until sync completes. If interrupted, next open resumes from the last successfully completed batch.

### 3. Implementation pattern
- **Key technologies / APIs**: Firestore `getDocs` with `orderBy("timestamp", "asc")` and `startAfter` cursor for pagination. IndexedDB batch writes inside a transaction.
- **Core code sketch** (minimal – simplified for v1 without pagination; full version would add `startAfter`):
  ```js
  async function firstInstallSync(onProgress) {
    const q = query(collection(db, "orders"), where("shopId", "==", SHOP_ID), orderBy("timestamp", "asc"));
    const snapshot = await getDocs(q);
    const orders = snapshot.docs.map(doc => ({ orderId: doc.id, ...doc.data() }));
    const db = await openDB();
    const tx = db.transaction(["orders", "syncMeta"], "readwrite");
    const ordersStore = tx.objectStore("orders");
    for (const order of orders) {
      await ordersStore.put(order);
    }
    const lastTimestamp = orders.length > 0 ? orders[orders.length - 1].timestamp : null;
    const syncStore = tx.objectStore("syncMeta");
    await syncStore.put({ id: "sync", lastSyncTimestamp: lastTimestamp, firstSyncComplete: true, lastSyncDate: new Date().toISOString() });
    await new Promise(resolve => { tx.oncomplete = resolve; });
    if (onProgress) onProgress(orders.length, orders.length);
    return orders.length;
  }
  ```
- **File structure / hooks**:
  - `db/syncService.js` – `firstInstallSync`.
  - `hooks/useSync.js` – checks `firstSyncComplete` and calls `firstInstallSync` with progress callback.
  - `owner/ShopRecord.jsx` – shows loading screen while `syncing` and `!firstSyncDone`.

### 4. Setup commands (if any)
- **Install**: No extra packages; uses existing Firebase and IndexedDB.
- **Initialisation**: IndexedDB must have `orders` store with `timestamp` index and `syncMeta` store.
- **Browser/API permissions**: None.

### 5. Known limits and edge cases
- **What it does NOT handle**:  
  - The current implementation in v1 fetches **all orders in a single query** (no batching). If order volume exceeds 500, the query may hit Firestore limits or timeout. The template mentions batching, but the actual code omitted it.  
  - If the daily read budget is exhausted mid‑sync, the sync will fail and leave `firstSyncComplete` false; it will retry on next open.  
  - No resumable batching in v1; if interrupted, it starts over.
- **Fails gracefully**: If sync fails (e.g., network error), the loading screen remains; on next app open, it retries from scratch. No partial data is committed because the transaction commits only at the end of the whole fetch (in v1). With proper batching, each batch is atomic, so partial progress would be saved.
- **Scalability ceilings**: Works for up to ~500 orders out‑of‑the‑box. For larger datasets, batching must be implemented (using `startAfter` cursor and `limit(500)`).

- **Must be present before use**:
